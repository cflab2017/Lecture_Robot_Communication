# 부록 C. 확인 문제 해답

> 각 챕터 "❓ 확인 문제"의 모범답안입니다. 먼저 스스로 풀고, 막히면 해당 챕터의 실습을 다시 따라 해 보세요.
> 코드로 검증 가능한 문제는 `lecture/code/`의 도구로 직접 확인하길 권합니다.

## 01. 환경 준비

**1. 실제 로봇은 Modbus TCP에 몇 번 포트를 쓰며, 실습에서 왜 1502를 쓰는가?**
실제 장비는 표준 포트 **502**를 사용합니다. 그러나 Windows에서 502 같은 1024 미만 특권 포트는 관리자 권한이 필요할 수 있어, 실습에서는 권한 문제 없는 **1502**(1024 이상)를 사용합니다. 명령마다 `--port 1502`를 붙이는 것을 잊지 마세요.

**2. pymodbus 버전을 3.6.9로 고정하는 이유는?**
pymodbus는 메이저/마이너 버전마다 함수 시그니처와 import 경로가 크게 바뀝니다(예: `slave=` vs `unit=`, 서버 API 변경). 교안 코드는 **3.6.9** 기준으로 작성·검증되었으므로, 다른 버전에서는 일부 호출이 실패할 수 있습니다.

**3. `300.000 mm`가 Low=-27680, High=4로 나뉘는데 Low가 음수인 이유는?**
`300000 = 4 × 65536 + 37856`에서 나머지 37856이 16bit signed 최대값(32767)을 넘습니다. signed 16bit로 표현하면 `37856 − 65536 = −27680`이 되어 Low word가 음수가 됩니다. (자세한 원리는 [부록 B](word-conversion.md))

## 02. I/O 통신

**1. 6축 로봇에서 RI/O의 입력·출력 채널 수와 사용 불가 모델은?**
입력 6개 / 출력 4개(**6I/4O**). **SCARA와 Delta** 로봇에서는 사용할 수 없습니다(로봇 본체 말단 배선이 없음). 대안으로 DI/O 또는 MI/O를 씁니다.

**2. FI/O 기본 매핑에서 `$SI[1]`, `$SO[3]`의 기능은?**
`$SI[1]` = **Start**(프로그램 시작 입력), `$SO[3]` = **Fault**(이상/고장 출력).
(입력 $SI: Start/Hold/Stop/Enable/RSR1~4, 출력 $SO: Run/Held/Fault/Ready/ACK1~4)

**3. 픽앤플레이스에서 P2로 갈 때 `PTP`가 아닌 `LIN`을 쓰고, P1을 P2 위 50mm에 두는 이유는?**
워크피스에 **수직으로 정밀 접근/이탈**하기 위함입니다. `LIN`(직선보간)은 경로가 직선으로 보장되어 옆면 충돌·끌림을 막고, 접근점 P1(P2 위 +50mm)을 경유하면 그리퍼가 위에서 수직으로 내려와 안전하게 집을 수 있습니다. `PTP`는 경로를 보장하지 않아 근접 구간에 부적합합니다.

## 03. Ethernet TCP/IP

**1. 로봇 IP가 `192.168.1.10`일 때 통신 가능한 IP는? (192.168.1.50 / 192.168.2.50 / 10.0.1.10 / 192.168.1.200)**
정답: **192.168.1.50, 192.168.1.200**. 앞 3옥텟 `192.168.1`이 일치해야 같은 IP 도메인입니다. `192.168.2.50`은 셋째 옥텟이, `10.0.1.10`은 첫째 옥텟부터 달라 통신 불가합니다.

**2. HRSS 코드 빈칸 채우기 (`COPEN(___, HANDLE)` / 버퍼 비우기 / RS-232의 첫 인자)**
① `ETH` (TCP/IP)  ② `CCLEAR` (버퍼 클리어)  ③ `SER` (RS-232).

**3. 값 `123`을 전송하면 상대가 받는 실제 문자열과 그 이유는?**
**`{123}`**. 패킷은 전송 시 시작/끝에 중괄호 `{` `}`가 자동으로 붙어 `{data}` 형태로 전달되기 때문입니다.

## 04. RS-232 직렬 통신

**1. RS-232 직접 배선에서 로봇 TX는 무엇에 연결하며, 함께 연결할 신호선은?**
로봇 **송신핀(TX)** 은 호스트의 **수신핀(RX)** 에, 로봇 RX는 호스트 TX에 연결합니다(서로 엇갈리는 **교차/크로스 배선**, 매뉴얼의 *twisted wires*). 그리고 양측 **공통 접지(Common GND)** 를 함께 연결합니다.

**2. `COPEN`의 첫 인자는 TCP/IP와 RS-232에서 각각 무엇이며, CWRITE/CREAD·패킷 포맷은 동일한가?**
첫 인자는 TCP/IP=**`ETH`**, RS-232=**`SER`**. `CWRITE`/`CREAD` 명령과 패킷 포맷(`{ }`, `,` 구분, non-format=ASCII decimal)은 **동일**합니다. 즉 물리 계층과 `COPEN` 첫 인자만 다릅니다.

**3. 한쪽 baud 9600, 다른 쪽 19200이면 통신되는가?**
**불가**. RS-232는 이더넷과 달리 속도 자동 협상이 없어 baud가 다르면 비트 타이밍이 어긋나 글자가 깨지거나 빈 값(`b''`)이 수신됩니다. 양측 baud rate를 **동일**하게(+데이터비트·패리티·정지비트도, 기본 8N1) 맞춰야 합니다.

## 05. MODBUS 기초

**1. RTU vs TCP를 (가)물리계층, (나)에러검증으로 비교하고, HIWIN의 RTU 인터페이스는?**
(가) 물리계층 — RTU는 시리얼(HIWIN은 **RS485**), TCP는 **Ethernet**. (나) 에러검증 — RTU는 데이터 끝에 **CRC 필수**, TCP는 하위 **TCP 계층이 무결성 검증**하므로 Modbus CRC 생략. HIWIN 로봇의 RTU는 **RS485**를 사용합니다.

**2. 다음 작업의 Function Code는? (SO 읽기 / Holding 단일 쓰기 / Coil 단일 ON / Input Register 읽기)**
**02** (Read Discrete Input) / **06** (Write Single Holding) / **05** (Write Single Coil) / **04** (Read Input Register).

**3. Input Register에 쓰기가 실패하는 이유와, 마스터가 쓸 수 있는 레지스터 2종은?**
**Input Register는 읽기 전용(R)** 으로, 슬레이브가 측정·상태값을 채우는 영역이라 마스터가 덮어쓸 수 없습니다(읽기 전용에는 쓰기 Function Code 자체가 없음). 마스터가 쓸 수 있는 R/W 레지스터는 **Coil**(05/15)과 **Holding Register**(06/16)입니다.

## 06. MODBUS SERVER (로봇=슬레이브)

**1. 마스터가 로봇을 명령/감시할 때 사용하는 레지스터 영역은?**
**명령**(쓰기)은 R/W 영역인 **Coil**(SI/DO 비트)과 **Holding Register**(명령 번호·파라미터·트리거)에 씁니다. **감시**(읽기)는 읽기 전용인 **Discrete Input**(SO/DI 상태)과 **Input Register**(동작 상태·현재 위치·명령 상태)를 읽습니다.

**2. 단일 명령 실행 5단계와, HR200을 0으로 리셋하지 않으면 생기는 문제는?**
① HR201←명령번호 ② HR202~←파라미터 ③ HR200←1(실행) ④ IR524가 2(Run)→1(Idle)·IR200=1(Success) 확인 ⑤ HR200←0(리셋). 마지막에 리셋하지 않으면 트리거가 1로 남아 로봇이 다음 명령을 받지 못하고 **사이클이 잠깁니다**.

**3. Z=+90.000mm를 LIN으로 보낼 때 Low/High word와 `& 0xFFFF`가 필요한 이유는?**
`90.000×1000 = 90000` → `split_word(90000) = (24464, 1)`. Modbus 쓰기는 unsigned 16bit만 받으므로 각 word를 **`& 0xFFFF`** 로 인코딩해야 합니다. +90mm는 두 word가 양수라 값이 안 바뀌지만, 예컨대 **−90.000mm = (40146, −2)** 처럼 High word가 음수면 `−2 & 0xFFFF = 65534` 변환이 **반드시** 필요합니다(없으면 `struct.error`). 교안의 `RobotMaster.write_holding`은 이를 자동 처리합니다.

## 07. MODBUS CLIENT (로봇=마스터)

**1. 로봇이 Master Client일 때 명령/감시에 쓰는 영역·명령은? 05장과 어떻게 대비되는가?**
**명령**은 R/W 영역인 **Coil**(`MBC_WRITE_COIL`)·**Holding Register**(`MBC_WRITE_HOLDING`)에 씁니다(예: 그리퍼 `HR1600=1` "열기"). **감시**는 R 영역인 **Input Register**(`MBC_READ_INPUT`)·**Discrete Input**(`MBC_READ_DINPUT`)을 읽습니다. 05장은 로봇이 명령을 **받는**(Server) 쪽, 06장은 로봇이 상대 장비에 **보내는**(Client) 쪽 — 같은 4종 레지스터를 반대 방향으로 씁니다.

**2. `'W'`와 `'D'`의 차이, HR204=24464/HR205=1을 `'D'`로 읽으면?**
`'W'`(Word)=16bit 1개 그대로. `'D'`(Double Word)=연속 2개를 32bit로 합치며 **Low word 먼저, High word 나중**. `High×65536 + Low = 1×65536 + 24464 = 90000`(0.001 스케일이면 90.000°). 16bit를 넘는 좌표·각도에 `'D'`를 씁니다.

**3. 실수 `10.5`를 Modbus로 보내려면?**
(a) Modbus 레지스터는 16bit 정수만 실어 나르므로 부동소수점을 그대로 못 넣습니다. (b) `IEEE754_ENCODE(10.5)=1093140480`(32bit 정수)로 인코딩해 `'D'`로 전송하고, 받는 쪽은 `IEEE754_DECODE(1093140480)=10.5`로 복원합니다. (c) 상대 장비의 word 순서가 반대면 `SWAP_WORD`로 상·하위 word를 교환해 보정합니다.

## 08. 모니터링 & 디버깅

**1. Wireshark에서 Modbus 패킷이 안 보일 때 두 가지 점검은?**
(a) 일반 랜카드를 캡처 중일 수 있음 → 인터페이스를 **"Adapter for loopback traffic"** 으로 변경(Npcap 미설치면 재설치). (b) 비표준 포트 1502는 자동 해석 안 됨 → 패킷 우클릭 **Decode As…** → 포트 1502를 **Modbus/TCP** 로 지정하면 Protocol 컬럼이 Modbus/TCP로 바뀝니다.

**2. 응답 `04 04 00 5a 00 00`을 해독하면?**
(a) FC **04h** = Read Input Registers. (b) 두 번째 `04`는 데이터 byte 수 = 개수×2 = 2×2 = 4. (c) `00 5a`=0x005A=**90**, `00 00`=**0**. (d) Word 3개를 읽었다면 byte 수는 3×2 = **6**.

**3. HR 201부터 111/222/333을 쓰는 요청·응답은?**
(a) 값이 2개 이상이므로 **10h**(Write Multiple Registers). (b) `10 00 C9 00 03 06 00 6F 00 DE 01 4D` (주소 0x00C9=201, 개수 3, byte수 6, 111/222/333). (c) 응답 `10 00 C9 00 03` — 쓰기 응답은 **시작주소+개수만 에코**하고 데이터는 되돌리지 않습니다(대역폭 절약).

## 09. 종합 실습

**1. 비전이 준 좌표(예: X=123.45mm)를 로봇에 LIN으로 보내기까지 거치는 변환은?**
ASCII 문자열 `{123.45,...}` → 파싱해 실수 `123.45` → `round(123.45×1000)=123450` → `split_word(123450)`로 Low/High word → Holding Register에 적재 → 명령 사이클 실행. (음수 좌표도 `write_holding`이 자동 인코딩)

**2. 그리퍼 동작 완료를 `IR[769]==2`만으로 판정하면 안 되는 이유는?**
닫기 동작 후 `IR769`는 2(Pos)로 **남아 있습니다**. 다음 열기 동작에서 곧바로 폴링하면 그리퍼가 움직이기도 전에 조건이 통과되어 버립니다. 따라서 **`IR[769]==2` 그리고 `HR[1606]==0`**(실행 플래그 자동 리셋)을 함께 확인해야 합니다. 로봇의 `HR[200]=0` 리셋과 같은 "상태 잔류(state residue)" 함정입니다.

**3. 한 사이클을 마치고 다음 부품을 처리하려면 무엇을 초기화해야 하는가?**
로봇은 매 명령 후 `HR[200]=0`으로 트리거를 리셋해야 다음 명령을 받습니다. 그리퍼는 `HR[1606]`이 0으로 자동 복귀했는지 확인합니다. 좌표·방향 등 파라미터는 새 값으로 덮어쓰면 됩니다.
