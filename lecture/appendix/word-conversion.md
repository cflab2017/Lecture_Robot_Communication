## 1. 왜 분할이 필요한가 (5분)

Modbus 레지스터 **1개 = 16bit signed**, 즉 표현 범위가 **-32768 ~ 32767** 뿐입니다.
최상위 비트(2^15)가 부호로 쓰이기 때문입니다.

그런데 로봇이 다루는 값은 이 범위를 쉽게 넘습니다.

| 값 | 스케일 | raw 정수 | 16bit 범위? |
|----|--------|----------|-------------|
| 90.830° (각도) | ×0.001 | 90,830 | 초과 ❌ |
| 300.000 mm (좌표) | ×0.001 | 300,000 | 초과 ❌ |
| -399.958 mm | ×0.001 | -399,958 | 초과 ❌ |

좌표·각도는 소수점을 살리려고 **0.001 스케일(정수화)** 을 쓰므로 값이 더 커집니다.
그래서 32bit 값을 **레지스터 2개(Low word + High word)** 에 나눠 저장합니다.
2개를 쓰면 범위가 **-2,147,483,648 ~ 2,147,483,647 (signed 32bit)** 로 넓어집니다.

> 저장 순서: Modbus의 `'D'`(Double word) 타입은 **Low word 먼저, High word 나중**입니다.

## 2. 복원 공식 (Low/High word → 실제값) — 매뉴얼 Appendix II

읽어온 두 레지스터에서 실제값을 복원하는 공식입니다.

> **값 = High × 65536 + (Low를 unsigned로 본 값)**
> 즉 Low가 음수면 `Low + 65536` 을 더한다.

매뉴얼은 "Low가 양수면 `65536×High + Low`, Low가 음수면 `65536×(High+1) + Low`"로 표현하는데,
둘은 같은 식입니다(음수 Low에 65536을 더하는 것 = High를 1 올리는 것).

**매뉴얼 예제 손계산**

| 항목 | Low | High | 손계산 | raw | ×0.001 |
|------|-----|------|--------|-----|--------|
| A1 각도 (Low 양수) | 25294 | 1 | 65536×1 + 25294 | 90830 | **90.830°** |
| X 좌표 (Low 음수) | -6742 | -7 | 65536×(-7+1) + (-6742) = 65536×(-6) − 6742 | -399958 | **-399.958 mm** |

A1 풀이: `65536 × 1 = 65536`, `65536 + 25294 = 90830`, `90830 × 0.001 = 90.830°`
X 풀이: Low가 음수이므로 High를 +1 → `65536 × (-6) = -393216`, `-393216 + (-6742) = -399958`, `× 0.001 = -399.958 mm`

**코드 검증**

```bash
>>> combine_word(25294, 1)      # 90830  ✅
>>> combine_word(-6742, -7)     # -399958  ✅
>>> to_engineering(-6742, -7)   # -399.958  (scale=0.001 자동 적용)
```

## 3. 분할 공식 (실제값 → Low/High word)

쓰기 전에 실제값을 두 레지스터로 쪼개는 과정입니다.

> **값 = 몫 × 65536 + 나머지** (나머지는 항상 0~65535)
> **High = 몫**, **Low = 나머지** (단, 나머지가 32767 초과면 `나머지 − 65536`)

핵심은 **음수일 때 몫을 "내림(floor)"** 한다는 것 — 나머지를 항상 양수로 만들기 위해서입니다.

**예제 1: +300.000 mm**

| 단계 | 계산 |
|------|------|
| raw | 300.000 × 1000 = **300000** |
| 분해 | 300000 = **4** × 65536 + **37856** |
| High | 몫 = **4** |
| Low | 37856 > 32767 → 37856 − 65536 = **-27680** |

**예제 2: -300.000 mm**

| 단계 | 계산 |
|------|------|
| raw | -300.000 × 1000 = **-300000** |
| 분해 | -300000 = **-5** × 65536 + **27680** (몫을 내림: -300000/65536 ≈ -4.58 → -5) |
| High | 몫 = **-5** |
| Low | 27680 < 32767 → 그대로 **27680** |

검산: `-5 × 65536 = -327680`, `-327680 + 27680 = -300000` ✅

**코드 검증**

```bash
>>> split_word(300000)     # (-27680, 4)   ← (low, high). Low가 음수!
>>> split_word(-300000)    # (27680, -5)   ← High가 음수
>>> split_word(90000)      # (24464, 1)    ← 90.000° (매뉴얼 read 예제)
```

> ⚠️ **실전 함정 — High word가 음수일 때**
> 위 -300mm 예처럼 High가 **-5** 같은 음수가 나오면, Modbus로 쓰기 전 레지스터는 *unsigned 16bit* 칸이므로
> **`& 0xFFFF`** 로 unsigned 인코딩해야 합니다 (`-5 & 0xFFFF = 65531 = 0xFFFB`).
> 강의의 `RobotMaster.write_holding`(06장)이 이 변환을 **자동 처리**하므로 직접 신경 쓸 일은 적지만,
> 패킷을 손으로 뜯어볼 때 음수가 `0xFFFB`로 보이는 이유를 알아두세요.

## 4. SWAP_WORD (5.4.4.1)

32bit 값의 **상위 word ↔ 하위 word** 를 통째로 맞바꿉니다.
장비마다 High/Low word 정의(byte/word order)가 반대일 수 있어, 다른 장비와 맞출 때 사용합니다.

**손계산: `swap_word(10)`**

```
입력 10  =  0x0000_000A
            └High word┘└Low word┘
            (0x0000)    (0x000A)

교환 후  =  0x000A_0000   =  10 × 65536  =  655360
```

| 입력 | 16진수 | 교환 후 16진수 | 결과 |
|------|--------|----------------|------|
| 10 | 0x0000000A | 0x000A0000 | **655360** |

**코드 검증**

```bash
>>> swap_word(10)    # 655360  ✅  (매뉴얼 Example 1과 동일)
```

## 5. IEEE754 단정도(single precision) (5.4.4.2 / 5.4.4.3)

Modbus는 **실수(float)를 직접 전송하지 못합니다.** 그래서 실수를 **IEEE754 단정도(32bit) 정수**로
인코딩해서 보내고, 받는 쪽에서 디코딩합니다. 32bit이므로 `'D'`(Double word)로 **레지스터 2개**에 실어 보냅니다.

```
encode:  10.5 (float)  ──IEEE754──▶  1093140480 (32bit int)  ──▶  Modbus 'D' 전송
decode:  1093140480 (int)  ──IEEE754──▶  10.5 (float)
```

**코드 검증**

```bash
>>> ieee754_encode(10.5)        # 1093140480  ✅  (매뉴얼 Example과 동일)
>>> ieee754_decode(1093140480)  # 10.5        ✅  (역변환)
```

> 분할(2~3장)과 IEEE754는 **별개의 표현법**입니다.
> - **정수 스케일 분할**: 좌표·각도처럼 "×1000 정수"로 보내는 방식(소수 자릿수 고정).
> - **IEEE754**: 임의 실수를 비트 그대로 보내는 방식(매우 크거나 작은 값도 표현).
> 어느 쪽을 쓸지는 상대 장비의 데이터 타입 규약을 따릅니다.

## 6. 직접 풀어보기 (연습문제 5개)

먼저 손으로 풀고, 아래 안내대로 코드로 검증하세요. 정답은 맨 아래 토글에 있습니다.

1. **복원**: Low word = 0, High word = 2 일 때 실제값(raw 정수)은? (`combine_word(0, 2)`)
2. **복원→공학값**: Low word = 24464, High word = 1 (각도, ×0.001) → 실제 각도(°)는? (`combine_word(24464, 1)`)
3. **분할**: 좌표 **+200.000 mm** 를 Low/High word로 분할하면? (raw = 200000)
   - 검증: `python word_tools.py 200`
4. **분할(음수)**: 좌표 **-150.000 mm** 를 Low/High word로 분할하면? (raw = -150000)
   - 검증: `python word_tools.py -150`
5. **IEEE754**: 실수 **3.14** 를 IEEE754 32bit 정수로 인코딩하면? 그리고 그 정수를 다시 decode하면?
   - 검증: `python word_tools.py 3.14` (출력의 `IEEE754` 줄)

> 각 문제는 `python word_tools.py <값>` 또는 파이썬 인터프리터에서
> `from word_tools import *` 후 함수를 직접 호출해 확인할 수 있습니다.

<details>
<summary>정답 보기</summary>

| # | 문제 | 손계산 | 정답 | 검증 명령 |
|---|------|--------|------|-----------|
| 1 | combine(0, 2) | 65536×2 + 0 | **131072** | `combine_word(0, 2)` |
| 2 | combine(24464, 1)×0.001 | (65536×1 + 24464)=90000, ×0.001 | **90.000°** | `to_engineering(24464, 1)` |
| 3 | split +200mm (raw 200000) | 200000 = 3×65536 + 3392 → 3392<32767 | **Low=3392, High=3** | `split_word(200000)` |
| 4 | split -150mm (raw -150000) | -150000 = -3×65536 + 46608 → 46608>32767, 46608-65536=-18928 | **Low=-18928, High=-3** | `split_word(-150000)` |
| 5 | encode 3.14 | IEEE754 single | encode=**1078523331**, decode≈**3.1400001** | `python word_tools.py 3.14` |

> 5번 decode가 3.14가 아니라 3.1400001인 것은 정상입니다 — 단정도 float의 정밀도 한계(반올림 오차)입니다.

</details>
